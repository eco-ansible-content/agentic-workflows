# Insight: Provider Auto-Detection Collision Pattern

**Date**: 2026-06-02
**Category**: Pattern
**Subcategory**: Provider-Auto-Detection
**Applies To**: Modules with pluggable provider/backend systems and auto-detection
**Applied To Agents**: module-worker, enhancement-specialist, ci-validation-specialist
**Severity**: High
**Source Epic**: ACA-6275
**Source Collection**: ansible.windows (PR #905)

## The Problem

When enhancing an existing module (win_package) with a new provider (package_management), the new provider was added to the provider registry (`$providerInfo` ordered hashtable). The module's auto-detection logic iterates ALL registered providers when `provider=auto` (the default), calling each provider's `Test` function to determine what type of package is installed.

The new `package_management` provider's Test function required the `package_management_provider` parameter (NuGet or PowerShellGet) to be set. When auto-detection called this Test function, the parameter was null, causing the module to fail with:

```
"package_management_provider is required when using the package_management provider"
```

This broke ALL existing win_package tests that used the default `provider=auto`.

## Root Cause Analysis

**What happened**:
1. New provider registered in `$providerInfo` dictionary
2. Auto-detection loop iterated all keys including the new provider
3. New provider's Test function required parameters not available during auto-detection
4. Module crashed with validation error before reaching the intended provider

**Why it happened**:
- The developer (agent) treated the new provider like existing providers
- Existing providers (msi, msix, registry) only need `path` and `product_id` for detection
- The new provider needed `package_management_provider` (a parameter unique to it)
- No awareness that auto-detection would call ALL providers including the new one

**Was it predictable?**
YES - Reading the auto-detection loop code would have revealed that all provider keys are iterated.

**Knowledge gap**:
- When adding a provider to a multi-provider module, always trace the auto-detection code path
- Providers requiring extra mandatory parameters cannot participate in auto-detection

## The Solution

Exclude providers that require explicit opt-in from the auto-detection list:

```powershell
# BEFORE (broken): All providers included in auto-detection
$providerList = [String[]]$providerInfo.Keys

# AFTER (fixed): Exclude providers requiring explicit opt-in
$providerList = [String[]]$providerInfo.Keys | Where-Object {
    ($_ -ne 'msix' -or $msixAvailable) -and $_ -ne 'package_management'
}
```

Note: The `msix` provider already had a conditional exclusion (`-or $msixAvailable`), which was a hint that this pattern existed.

## Generalized Pattern

**When adding a new backend/provider to an existing module**:

1. **Trace the detection path**: Find where the module iterates providers automatically
2. **Check parameter requirements**: Does the new provider need parameters beyond the base spec?
3. **Apply exclusion rule**: If yes, exclude from auto-detection and require explicit opt-in
4. **Test existing behavior**: Run ALL existing tests with default parameters BEFORE pushing
5. **Document the exclusion**: Comment explaining why the provider is excluded

**Code pattern** (generalized):

```python
# Python equivalent
providers_for_auto_detect = [
    p for p in registered_providers
    if p.can_detect_without_extra_params()
]
```

```powershell
# PowerShell equivalent
$autoDetectProviders = $allProviders.Keys | Where-Object {
    $allProviders[$_].RequiresExplicitOptIn -ne $true
}
```

## Prevention Checklist

When enhancing a module with a new provider/backend:

- [ ] Read the auto-detection/auto-selection code path
- [ ] Identify what parameters the detection function needs
- [ ] If detection needs provider-specific parameters: exclude from auto-detection
- [ ] Run existing module tests with DEFAULT parameters before committing
- [ ] Add explicit provider selection to all tests for the new provider
- [ ] Comment the exclusion with rationale

## Impact

**Before**: 
- CI failure on ALL existing win_package tests
- 1 extra commit + CI re-run cycle required

**After**:
- Pattern documented for future provider additions
- Enhancement specialist guided to check auto-detection paths
- Saves 30-60 minutes per future provider enhancement
