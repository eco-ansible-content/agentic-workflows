# Git Workflows - Smart Strategy for New vs Enhancement Projects

The ansible-collection-swarm uses **two completely different git workflows** based on whether you're building a new collection or enhancing an existing one.

## TL;DR

| Mode | Location | Git Workflow | Philosophy |
|------|----------|--------------|------------|
| **New Project** | `~/agentic-workflow-collections/` | Direct push to main | You own it completely |
| **Enhancement** | Your cloned repo | Branch → Fork → PR | Collaborative workflow |

---

## Workflow 1: New Projects (Full Build)

### When This Happens

- Epic describes a **new** collection that doesn't exist yet
- No existing `galaxy.yml` found anywhere
- Building from scratch

### Philosophy

**Complete autonomous control.** This is your temporary workspace. Push directly to main - no branches, no PRs, no ceremony. Work is never distractive when using git right.

### Location

```
~/agentic-workflow-collections/<namespace>/<name>/
```

This is a **temporary sandbox** for autonomous builds. Not your production repo.

### Git Operations

```bash
# 1. Swarm builds everything in workspace
cd ~/agentic-workflow-collections/microsoft/scvmm/

# 2. Commit with quality message
git commit -m "Complete Ansible collection: microsoft.scvmm

Implemented from Jira Epic: EPIC-2345

Modules:
- scvmm_vm - Manage virtual machines
- scvmm_host - Manage Hyper-V hosts
- scvmm_cloud - Manage private clouds

Testing:
- All integration tests passing
- Test environment: 192.168.50.10 (SCVMM 2022)

Generated by: Agentic Workflows (ansible-collection-swarm)
Agent run: 2024-05-28"

# 3. Push directly to main (if git delivery selected)
git push -u origin main
```

**Rules**:
- ✅ Push directly to main
- ✅ Complete control
- ✅ Fast autonomous delivery
- ❌ Don't destroy test environment
- ❌ Don't force push

### What You Do After

1. Review the generated collection
2. Test it yourself
3. When satisfied:
   - Option A: Use it as-is from this workspace
   - Option B: Move to your production repo
   - Option C: Archive and deploy via CI/CD

## Workflow 2: Enhancements (Existing Projects)

### When This Happens

- Collection already exists in:
  - Current directory (you're in a cloned repo)
  - Swarm workspace (previous build)
  - Ansible collections path (installed via ansible-galaxy)
  - Custom location you specify
- Adding modules to existing collection

### Philosophy

**Collaborative workflow.** This is a real project with other developers. Respect the workflow. Use branches, forks, and PRs. Let code review agents and CI validate before merge.

### Location

```
/Users/you/projects/ansible-collections/microsoft-scvmm/  # Your cloned repo
```

Or wherever you've cloned/forked the collection.

### Git Operations

```bash
# 1. Update main before starting
cd /Users/you/projects/ansible-collections/microsoft-scvmm/
git checkout main
git pull origin main

# 2. Create feature branch
git checkout -b add-modules-epic-5678

# 3. Swarm implements new modules

# 4. Commit with quality message
git commit -m "Add modules from EPIC-5678

Modules added:
- scvmm_template - Manage VM templates
- scvmm_library - Manage library shares

Changes:
- Implemented 2 new modules
- Added integration tests for all modules
- Updated module backlog
- All tests passing (new + regression)

Epic: https://jira.company.com/browse/EPIC-5678
Test environment: 192.168.50.10 (SCVMM 2022)

Co-authored-by: Agentic Workflows <noreply@eco-ansible-content.com>"

# 5. Push to YOUR fork (not origin!)
git push fork add-modules-epic-5678

# 6. Create Pull Request
gh pr create \
  --title "Add modules from EPIC-5678" \
  --body "..." \
  --base main \
  --head fork:add-modules-epic-5678
```

**Rules**:
- ✅ MUST update main before branching
- ✅ MUST use feature branch (never commit to main)
- ✅ MUST push to fork (never `git push origin main`)
- ✅ MUST create PR
- ✅ Quality commit message
- ❌ NEVER push directly to origin
- ❌ NEVER force push
- ❌ NEVER skip regression tests

### Prerequisites

You MUST have a fork remote configured:

```bash
# Check if fork remote exists
git remote -v

# If not, add it
git remote add fork https://github.com/YOUR-USERNAME/ansible-collections.git
```

The swarm will **error** if enhancement mode is detected but no fork remote exists.

### What Happens After PR

1. **Code Review Agents** - `superpowers:code-reviewer` validates code quality
2. **CI Pipeline** - Tests run (new modules + regression)
3. **Team Review** - Other developers review
4. **Merge** - PR merged to main when approved

## How The Swarm Decides Which Workflow

**Phase 0: Collection Detection**

The lead-architect checks multiple locations:

```bash
# Location 1: Current directory
if [ -f "./galaxy.yml" ]; then
  # Found existing collection → ENHANCEMENT MODE
  # Uses: Branch/Fork/PR workflow
fi

# Location 2: Swarm workspace
if [ -d "~/agentic-workflow-collections/$NS/$NAME/galaxy.yml" ]; then
  # Found existing collection → ENHANCEMENT MODE
fi

# Location 3: No collection found anywhere
# NEW COLLECTION → FULL BUILD MODE
# Uses: Direct push workflow
```

**The swarm tells you which mode**:

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📋 Git Workflow
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ ENHANCEMENT MODE

Git workflow: Branch + Fork + PR (collaborative)
  • Updates main before starting
  • Creates feature branch: add-modules-epic-5678
  • Commits with quality message
  • Pushes to your FORK (not origin)
  • Creates Pull Request automatically
  • CI and code-review agents validate via PR

Requirements:
  ⚠️  You MUST have a fork remote configured:
     git remote add fork <your-fork-url>
```

## Troubleshooting

### "ERROR: No fork remote found"

**Problem**: Enhancement mode detected but you don't have a fork configured.

**Solution**:
```bash
git remote add fork https://github.com/YOUR-USERNAME/repo.git
```

Then re-run the swarm.

### "Merge conflicts detected"

**Problem**: Main changed while you were working.

**Solution**:
```bash
git checkout main
git pull origin main
# Resolve conflicts
# Re-run swarm on updated main
```

### "How do I change from enhancement to full build mode?"

**Answer**: You can't - the mode is auto-detected. If you want full build mode but a collection exists, delete or rename the existing collection first.

## Configuration

Both workflows are **automatic** - no configuration needed. The swarm decides based on:

1. **Collection exists?** → Enhancement mode (fork/PR workflow)
2. **Collection new?** → Full build mode (direct push workflow)

The only requirement for enhancement mode:
```bash
# Must have fork remote configured
git remote add fork <your-fork-url>
```
